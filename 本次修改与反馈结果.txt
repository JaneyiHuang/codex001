# MAPPO Actor 剪枝后的知识蒸馏模块计划

## Summary
新增一个“剪枝后蒸馏”流程：以原始 MAPPO actor 作为 teacher，以剪枝后的 actor 作为 student，通过 teacher 在环境中采集到的真实观测 `obs` 训练 student 拟合 teacher 的动作概率分布。目标是让剪枝模型在保持参数量/推理速度优势的同时，性能更接近原始 MAPPO。

默认采用你确认的方案：

- 蒸馏方式：策略蒸馏，使用 KL 散度拟合 teacher/student logits 分布
- 数据来源：teacher MAPPO 在环境中 rollout 收集真实 obs
- 蒸馏对象：只蒸馏 actor，不蒸馏 critic，因为部署和推理决策只需要 actor

## Key Changes
- 新增 `distill_mappo_actor.py`
  - 加载 teacher：`results/mappo_checkpoint.pt`
  - 加载 student 初始模型：`results/mappo_actor_pruned_p25.pt`
  - 用 teacher 在 `MECEnv` 中运行若干 episode，收集所有 agent 的 obs
  - 用 teacher actor 输出 soft targets，训练 student actor
  - 保存蒸馏后的模型到：`results/mappo_actor_pruned_distilled_p25.pt`

- 蒸馏训练默认参数
  - `--collect-episodes 100`
  - `--epochs 30`
  - `--batch-size 512`
  - `--lr 1e-4`
  - `--temperature 2.0`
  - `--val-ratio 0.1`
  - 损失函数：`KL(student_logits / T, teacher_logits / T) * T^2`
  - 额外记录：train loss、val loss、teacher-student action agreement

- 修改统一对比入口 `compare_experiments.py`
  - 新增实验名：`distilled_mappo`
  - 新增参数：`--distilled-model-path`
  - 支持统一运行：
    ```powershell
    python compare_experiments.py --experiments mappo,pruned_mappo,distilled_mappo,offload,greedy --episodes 50
    ```

- 调整剪枝策略加载器
  - 将当前 `PrunedMAPPOPolicy` 改成可传入 `policy_name`
  - `pruned_mappo` 和 `distilled_mappo` 复用同一个压缩 actor 加载逻辑，只是模型路径和展示名称不同

## Assumptions
- “整流模块”这里按“知识蒸馏模块”理解，不额外加入 ReLU/激活函数结构改造。
- 第一版不做环境 reward 微调，只做离线策略蒸馏；这样结果更容易解释为“剪枝 + 蒸馏恢复策略性能”。

# 我按照你说的已经运行了，你觉得结果如何？


我看了你这次的 [comparison_loads_summary.csv](D:/ZJUT/miniPaper/Code/codex001/results/compare_experiments/comparison_loads_summary.csv) 和 [蒸馏日志](D:/ZJUT/miniPaper/Code/codex001/results/mappo_actor_pruned_distilled_p25_log.csv)，整体结果是**很好的**，尤其是蒸馏模块的效果很明确。

核心平均结果：

```text
方法              平均时延      平均丢弃率     平均卸载率
mappo             153.973      0.20005      0.78393
pruned_mappo      154.113      0.20735      0.76587
distilled_mappo   153.977      0.20014      0.78390
offload           166.401      0.07962      1.00000
greedy            179.497      0.57987      0.26598
```

我的判断：

**1. 蒸馏非常成功。**  
直接剪枝后，`pruned_mappo` 的平均时延只比原始 MAPPO 高 `0.14` 左右，但丢弃率从 `0.20005` 增到 `0.20735`。蒸馏后，`distilled_mappo` 几乎完全贴回原始 MAPPO：平均时延 `153.977` vs `153.973`，平均丢弃率 `0.20014` vs `0.20005`。这个结果很适合写成“蒸馏有效恢复剪枝导致的性能损失”。

**2. 负载扫描趋势也很漂亮。**  
13 个负载点里，`distilled_mappo` 和 `mappo` 的差距基本都在极小范围内，很多点甚至略好一点点。这说明蒸馏不是只在某一个固定负载下有效，而是在轻载到重载变化下都保持了策略行为。

**3. offload 的 drop rate 仍然最低，但时延明显更差。**  
这和之前一致：`offload` 平均丢弃率只有 `0.0796`，但平均时延 `166.40`，比 MAPPO / distilled MAPPO 高约 `12.4`。所以论文里可以说：全卸载能降低丢弃，但会牺牲长期平均时延；本文目标是最小化长期平均时延，MAPPO 系列在时延上更优。

**4. 蒸馏日志非常能支撑论文。**  
你的蒸馏训练 30 epoch 后：

```text
val_loss: 0.004038 -> 0.000074
val_agreement: 0.9944 -> 0.9987
```

这说明 student actor 和 teacher actor 的动作选择高度一致。这个可以作为剪枝蒸馏模块的一个辅助指标写进去。

论文里可以这样总结：

> Compared with the directly pruned actor, the distilled pruned actor effectively recovers the policy performance of the original MAPPO actor. Across different task load levels, the distilled model achieves nearly identical average delay and drop rate to the original MAPPO, while maintaining the reduced network size obtained by neuron pruning.
